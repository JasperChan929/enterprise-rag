**⚠️ 量化范围说明**:
本节量化的是 chunker 切断的 **1 种形态**: 表格切断 (利润表/收入表表头被切走,
数字行进入下一 chunk). chunker 至少还有 4 种切断形态没量化:
- 指代切断 ("本公司"无公司名)
- 时间锚点切断 ("报告期末"无年份)
- 口径说明切断 (数字 chunk 和口径定义 chunk 分离)
- 叙事切断 ("因此..." 但原因在前一 chunk)

这 4 种形态登记为 **TD-8-4** (见 6.2 节), 由 Day 9 量化. **TD-8-1 当前 P3
判定仅基于"表格切断"这 1/5 形态**, 全量量化后优先级可能需要重新评估.

**反直觉观察**: 招行碎片率 0% 看起来"干净", 但其实是 TD-5-1 (招行表格抽取
率 1%) 的副作用 — 表格没被 pdfplumber 识别成 table chunk, 被"文字化"揉进
text chunk 里连带数字一起, 所以不触发"含关键词无数字"的表格切断规则.
**碎片率低不等于数据质量好**.

## 3. 核心知识点

### 3.1 "4 层瓶颈模型" 的真正理解 (从 Day 7 到 Day 8 的认知升级)

Day 7 提出 4 层光谱 (L0 / L1 / L2 / L4), 作为瓶颈分析的框架. Day 8 实证它
之后发现, **这个框架本身是对的, 但是粒度不够**.

**为什么 Day 7 的"L2 LLM 作答"太笼统**:
Day 7 对 L2 的描述是 "金页进了 Top-K, 但 LLM 没抽出答案". 听起来清楚, 但
实际遇到时修法完全不明:
- 是加 Prompt 规则? → 如果 Top-5 证据本来就碎片, 加 Prompt 没用
- 是加 Reranker? → 如果 Top-5 证据已经完整, 加 Reranker 也没用

**L2 必须分子类型才能指导行动**:
L2a "LLM 认知问题":

症状: Top-5 里有自足 chunk (3 要素齐全的完整证据), LLM 仍拒答或答错
根因: LLM 对多口径/抽象词/隐式推理不熟练
典型: U4 招行 (Top-5 有 3 个不同口径的百分比, LLM 无法合并/选择)
修法: Prompt 工程 (低成本, Day 10 可做)

L2b "排序机械问题":

症状: Top-5 全是碎片 chunk, 但自足 chunk 在 Top-6~20
根因: RRF 融合后没把真金页推到前 5
典型: U3 国电南自 (自足 chunk 在 Rank 15/18)
修法: Reranker 条件开启 / 调 RRF k / 扩 top_k (中成本, Day 11+)


**核心诊断工具: "自足 chunk" 概念**

一个 chunk 是"自足"的, 当且仅当它同时包含 3 个要素:
要素 1: 主体 (哪家公司?)

由 metadata.company 保证, 不要求 chunk 文本里有公司名
⚠️ 前提: Prompt 必须把 metadata 拼进上下文让 LLM 看到
(如果 Prompt 没拼 metadata, 这个要素就失效 → TD-8-3)

要素 2: 时间锚点 (什么年?)

chunk 文本里有 "2025年" / "2024年" 等年份数字
或含 "报告期末" / "本期末" / "本报告期" (年报语境下等同于本年)
⚠️ Day 7 的 TD-7-1 错就错在只认"年份数字"不认"报告期末"

要素 3: 核心数据 (具体答案是什么?)

chunk 文本里有直接答案 (如百分比, 金额, 数字)
不能是"绕弯说"(如"稳健增长", "双位数")


三要素齐全 → LLM 一看就能答
缺 1-2 要素 → LLM 需要推理/组合, 可能失败

### 3.2 "非对称判定" 方法论 (证据推理的核心)

Day 8 审计的所有脚本都遵循一个共同原则: **不同方向的结论需要的证据强度不同**.

**对称判定 (错误方式)**:
Step 1 命中 → 数据存在
Step 1 未命中 → 数据不存在

这是对称判定, 看起来简洁, 但实际**假阴性代价远大于假阳性**. "未命中"
可能有很多原因:
- 关键词表不够宽 (毛利率 vs 毛利润率)
- PDF 抽取有字符错误
- chunker 把词切到两个 chunk
任一原因都会导致"数据其实存在但脚本说没有"的误判, 把一个可修的问题
(L0/L1/L2) 误判为不可修的 L4, **直接把优化方向引错**.

**非对称判定 (正确方式, 又名"宽进严出")**:
要排除 L4 (说"数据存在"): 一条证据就够 (PDF grep 命中任一页即可)
要定案 L4 (说"数据不存在"): 多路证据都阴性才定 (PDF + Qdrant 双阴性)
**原理**: 证据推理里的**默认立场**要向"数据可能存在"倾斜, 因为:
- 证伪 L4 的代价: 多跑一轮 Qdrant scroll (几秒钟)
- 错判 L4 的代价: 下游优化方向错 (几天工作量)

**扩展到其他场景**:
- 判 "用户行为异常" (风控): 宽进严出 — 异常证据要叠加多个维度才定案
- 判 "代码有 bug" (debug): 宽进严出 — 现象可疑时先收集多路证据, 不急于下结论
- 判 "LLM 输出是幻觉" (RAG): 宽进严出 — 一条证据不足, 要交叉源材料

**这个方法论是 Day 8 之外的长期资产**, 不只是今天这个审计脚本.



### 3.3 LLM 自述判定的根本缺陷

Day 4-7 的诊断失败率 3/4 = 75%, 根因是**依赖 LLM 自述作为诊断证据**.

**LLM 自述为什么不可靠**:

(1) **LLM 的"不确定"和"没有"表述几乎一样**
   - 真实情况 A: 年报没披露这个数据
   - 真实情况 B: 年报披露了但 Prompt 里候选 chunk 都没有这个数据
   - 真实情况 C: 候选 chunk 里有数据但 LLM 没识别出来
   
   三种情况下, LLM 输出都可能是 "参考资料未提及" 或 "无法回答".
   LLM 自述无法区分 A/B/C.

(2) **LLM 有拒答偏好** (尤其是 Prompt 里有"不够确定就拒答"规则时)
   - 看到多个口径的数据 → 选择拒答 (U4 招行)
   - 看到抽象词"情况"找不到精确对应 → 选择拒答 (U2 宁德)
   - 看到财务表头没数字 → 选择拒答 (U3 国电南自)
   
   拒答在 LLM 视角是"负责任", 在诊断视角是"信息丢失".

(3) **LLM 的自述语言缺乏结构化**
   - "未披露" / "未明确提供" / "无法确定" / "缺少相关数据" 语义上都相同
   - 工程师读这些话时, 自动把它们映射到同一个 bucket ("L4 数据缺失")
   - 但真实根因可能分散在 L0/L1/L2

**正确替代方案**: **用脚本证据替代 LLM 自述**.
脚本的判定是二元的 (命中/未命中), 可复现, 可审计. 这是 Day 8 做的事.

**面试可追问的深化**:
LLM-as-Judge 模式在 RAG 评估里很流行 (RAGAS 就是), 但 Day 8 的发现提醒我们:
**LLM-as-Judge 做诊断归因时很危险**. 它可以评估"答案质量好不好", 但不能评
估"问题出在系统哪一层". 层级归因必须用脚本证据.



### 3.4 脚本规则设计: 自我攻击方法论 (反面教材)

这一节记录**脚本 26 v1 被推翻再重写 v2** 的过程, 作为"脚本规则设计"的
反面教材.

**v1 的失败规则**:
```python
是碎片 chunk = 含财务关键词 AND 不含数字
```

看起来简洁直接, 但实际有**严重假阳性漏洞**:
假阳性示例:
"营业收入稳健增长, 净利润保持双位数增速"
↑ 含"营业收入""净利润"关键词, 也无数字, 按 v1 规则会被当成碎片
↑ 但这是叙述性文字, 不是表格切断!

跑下来, 6881 个 chunk 里会有大量 **"含财务关键词但本来就不该有数字"** 的叙述
chunk 被错误计数. 结果会严重虚高, 误导 Day 10 投入到一个不存在的问题上.

**为什么当初我会写出 v1**:
- 脑子里锚定了一个具体案例 (U3 p.138 "母公司利润表"), 只想抓住它
- 没做反向思考: **"这个规则还会抓到哪些非目标 chunk?"**
- 工程师直觉"这规则应该够用", 但直觉不等于证据

**v2 的修正**: 加入第 3 个条件 — **表头特征** (含"单位:"/"币种:"标识或
末尾列名列表), 把叙述 chunk 过滤掉. 跑完后 118 个潜在碎片被过滤到 25 个真碎片,
**79% 的假阳性噪声被滤除**.

**沉淀的方法论规则 (写进工作习惯)**:

> **🛑 脚本规则设计自我攻击清单** (写定义后、跑脚本前必做)
>
> 1. 列 3 个假阳性场景 (会被误抓的非目标 chunk)
> 2. 列 2 个假阴性场景 (会漏抓的目标 chunk)  
> 3. 对每种明确标注: 接受 / 加约束过滤
> 4. 预估结果范围: 如果跑出来偏离预估 2 倍以上, 规则有问题要重审
> 5. 把以上 4 项写进脚本注释顶部

**这是"工程师 vs 研究员"心态的分水岭**:
- 研究员心态: 有了一个规则, 先跑, 看数据, 有问题再改
- 工程师心态: 有了一个规则, 先自我攻击, 补漏洞, 然后跑 (减少返工)

Day 8 前半段我是研究员心态, 后半段被用户 (你自己) 识别出问题, 强制切换
到工程师心态. **这个切换本身是 Day 8 最值钱的认知升级之一**.






## 4. 关键实验发现

### 4.1 U1 比亚迪汽车业务毛利率 — 唯一 Day 7 判对的案例 (L4)

**Day 7 判定**: L4 数据缺失 (年报未按汽车业务单独披露成本)
**Day 8 审计**: L4 (一致)

**证据链**:
Step 1 PDF grep (002594_比亚迪_2025年年度报告.pdf, 268 页):
关键词: ['汽车业务毛利率', '汽车及相关产品毛利率', '汽车毛利率',
'汽车业务毛利', '分行业毛利率']
结果: ❌ 268 页全 PDF 无一命中
耗时: 102.5 秒Step 2 Qdrant scroll (company=比亚迪, year=2025, 共 728 chunks):
结果: ❌ 0 个 chunks 命中
耗时: 1.0 秒Step 3 Top-20 召回 (因 Step 2 为空, 跳过精确匹配):
Top-20 页面: [90, 47, 266, 245, 209, 16, 224, 204, 13, 15, ...]
观察: 这些页面里有财务数据相关页 (如 p.245 汽车业务营收), 但没一页讨论
"汽车业务毛利率" 这个口径.


**结论**: PDF grep + Qdrant 双阴性 → L4 真数据缺失 (年报只披露了集团合并
成本, 没有"汽车业务单独成本"数据, 毛利率算不出来).

**为什么 Day 7 这次判对了**: 运气 + 多次追问. Day 7 实际上用了 LLM 多次确认,
LLM 输出 "参考资料提到了汽车业务收入 648,646 百万元, 但未提供对应的营业
成本数据". 这段描述**恰好准确**, 所以 Day 7 判定正确.

**反过来说**: Day 7 对 U1 判对是**幸存者偏差**. 如果 LLM 这次说 "资料未提及",
Day 7 仍然会判 L4, 但判定依据会弱很多 (U2/U3 就是这样被误判的).

**面试可用**: 这个案例是"LLM 自述偶尔准确"的例子, 正好衬托出 U2/U3/U4 的
判定错误不是偶然 — 如果诊断方法依赖 LLM 自述, 准确率就是掷骰子.

### 4.2 U2 宁德磷酸铁锂产品 — 从 L4 推翻为 L2

**Day 7 判定**: L4 数据边界 (未披露磷酸铁锂专项数据)
**Day 8 审计**: L2 (推翻)

**证据链**:
Step 1 PDF grep (300750_宁德时代_2025年年度报告.pdf, 232 页):
✅ 命中 3 页:
p.14 - "公司动力电池产品包括电芯、模组/电箱及电池包。公司可提供磷酸铁锂
电池、三元高压中镍电池、三元高镍电池..."
p.17 - "2025年我国三元与磷酸铁锂正极材料合计产量达456.5万吨..."
p.21 - "二代神行超充电池是全球首款兼具800公里续航和峰值12C超充速度的
磷酸铁锂电池..."Step 2 Qdrant scroll (company=宁德时代, year=2025, 共 726 chunks):
✅ 命中 3 chunks (同 Step 1 的 3 页, content 完全对齐)Step 3 Top-20 召回 (query="宁德时代2025年磷酸铁锂电池产品情况"):
✅ Step 2 命中的 chunk 在 Top-20 首次出现 Rank 5
(Rank 5 和 Rank 16 都是金页)


**判定依据**: 三路全阳性 + Top-20 召回到 → 数据完整, 检索 OK → L2 (LLM 作答).

**Day 7 为什么判错**:
LLM 在这类查询上输出 "年报未做磷酸铁锂专项披露". 这段自述表面看像是 L4,
**实际 LLM 真实意图**是:
- 用户问 "磷酸铁锂电池**产品情况**"
- 年报写的是 "磷酸铁锂电池" + 具体产品名 (神行电池 / Pro 电池 / 钠新电池)
- LLM 觉得 "产品情况" 是个抽象词, 年报没有一段叫"磷酸铁锂产品情况"的章节
- 所以 LLM 说 "未做专项披露"

**真实根因**: LLM 对"抽象词"的对应能力弱. 它找不到字面对应的章节就倾向拒答,
而不是从具体产品信息里归纳出"产品情况". 这不是数据缺失, 是 **LLM 语义粒度
错配**, 属于 L2 的第 3 种子类型 (暂叫 L2c, 但今天不展开分类).

**Day 10 修法**:
- Multi-Query 改写成 "宁德磷酸铁锂电池的具体产品型号" → 用户抽象词翻成
  LLM 能找的具体词
- Prompt 加 "遇到抽象词先找具体对应, 不要直接拒答" 引导

### 4.3 U3 国电南自 2024 净利润 — 从 L4 推翻为 L2b (排序问题)

**Day 7 判定**: L4 数据链断裂 (推测 loader 丢了利润表)
**Day 8 审计**: **L2b** (推翻, 排序问题)

**证据链**:
Step 1 PDF grep (600268_国电南自_2024年年度报告.pdf, 288 页):
✅ 命中 21 页, 样例:
p.2 - "...本年度归属于上市公司股东净利润的比例为41.76%"
p.7 - "...扣非净利润 非经常性损益 经营活动..."
p.8 - "...中国会计准则披露的财务报告中净利润..."
(还有 p.138/p.139 的利润表主表相关页)Step 2 Qdrant scroll (company=国电南自, year=2024, 共 821 chunks):
✅ 命中 27 chunksStep 3 Top-20 召回 (query="国电南自2024年净利润"):
Top-20 页面: [251, 138, 126, 282, 80, 54, 174, 127, 8, 127, 245, ...]
自足 chunk (Top-20 里满足 3 要素的):
✅ Rank 15 (p.232) - 外汇风险附注里的净利润数据
✅ Rank 18 (p.258) - 利率风险附注里的净利润数据
Top-5 里自足 chunk: ❌ 无

**判定依据**: Top-5 全碎片, 自足 chunk 在 Rank 15/18 → L2b (排序问题).

**Top-5 为什么全是碎片** (关键发现):
Rank 1 (p.251, table): 子公司净利润表 (非母公司, 错口径)
Rank 2 (p.138, text):  "母公司利润表 2024年1—12月单位:元 币种:人民币"
↑ 这是 TD-8-1 表头切断样例, 真正数字在 p.139
↑ p.139 没进 Top-20
Rank 3 (p.126, text):  审计报告, 提 2024 但无具体数字
Rank 4 (p.282, table): 子公司投资变动表 (不相关)
Rank 5 (p.80,  text):  仲裁案件金额 (和净利润无关)


**Day 7 为什么判错**:
LLM 在这种情况下输出 "参考资料未提供 2024 年净利润". 看到这段自述, Day 7
推测 "loader 丢了利润表" → L4 数据链断裂.

实际上:
- Step 1 PDF grep 证明利润表在 PDF 里 (21 页含 "净利润")
- Step 2 Qdrant 证明利润表的某些 chunk 入库了 (27 个 chunks 命中)
- Step 3 证明召回池 (Top-20) 也有自足 chunk

**真实根因**: RRF 排序把真金页压在 Rank 15/18, Top-5 被子公司/附注类 chunk 占满.
这不是 L4 数据缺失, 是 **L2b 排序问题**.

**Day 10 修法**:
- Reranker 条件开启 (L2b 专门解药, Cross-Encoder 对"数字+主体"精信号更敏感)
- 或 top_k 从 5 扩到 10 (让 Rank 15 依然进不去, 但 Rank 6-10 里可能有其他候选)

### 4.4 U4 招行 2025 不良贷款率 — 从 L0 推翻为 L2a (多口径错位)

**Day 7 判定**: L0 chunking 边界 (TD-7-1 切走年份锚点)
**Day 8 审计**: **L2a** (推翻, LLM 认知问题)

**证据链 (最有戏剧性)**:
Step 1 PDF grep (600036_招商银行_2025年年度报告.pdf, 350 页):
✅ 命中 11 页, 多口径数据并存
Step 2 Qdrant scroll (company=招商银行, year=2025, 共 1388 chunks):
✅ 命中 14 chunks
Step 3 Top-20 召回:
⭐ Rank 1 (p.45): 本公司口径 0.93%, 自足!
⭐ Rank 3 (p.50): 消费信贷口径 1.52%, 自足!
⭐ Rank 5 (p.31): 本集团口径 0.94%, 自足!
Top-5 有 3 个自足 chunk → L2a 定案



**核心发现**: **Top-5 里有 3 个不同口径的不良贷款率数字**:
- 本公司 0.93%
- 本集团 0.94%  
- 消费信贷业务 1.52%

LLM 看到这 3 个数字**无法决定用户问的是哪个口径**, 选择拒答或给出含糊答案.
**这才是 U4 真正的问题, 不是 chunking 切走了年份**.

**Day 7 的 TD-7-1 错在哪**:
Day 7 发现 "p.45 chunk 里只写'报告期末 0.93%'没写'2025年'", 就推断
"chunker 把 2025 年锚点切走了, 导致 LLM 不确定是哪年的数据". 归因为 L0.

但 Day 8 证据显示:
1. Top-2 (p.5) 明确写了 "2025年度报告" + "2025年是很不平凡的一年"
   → LLM 能从 Top-5 其他 chunk 推断出 "本次查询的时间是 2025 年"
2. 真正让 LLM 卡住的不是 "哪一年", 是 "哪个口径"

**Day 7 混淆了 "现象" 和 "根因"**:
- 现象: p.45 chunk 文本里没 "2025 年" 字样 (这个观察**没错**)
- Day 7 误推根因: 所以 LLM 不知道年份, 所以拒答 (这个**错了**)
- 真实根因: Top-5 多口径并存, LLM 无法确定用户想要哪个

**Day 10 修法**:
- Prompt 加口径引导: "如遇多个口径 (本公司/集团/业务), 优先提供问题字面
  最匹配的口径, 并标注其他口径供参考"
- Prompt 拼 metadata: "本次查询公司=招商银行, 年份=2025" → LLM 不需要
  从 chunk 里推断时间主体

### 4.5 归纳: 4 翻车样例下的诊断偏差观察 (待大样本验证)

4 个案例 3 个推翻, 75% 错误率. 但这是 **4 翻车样例下的偏差观察, 不是统计学
意义的"系统性偏差"** — 4 样本量太小, 不足以下全局结论.

**⚠️ 样本边界**:
- 这 4 个案例 (U1-U4) 全部继承自 Day 4-7 的"已知拒答样例", 不是代表性采样
- 样本带强烈的"翻车驱动"偏向, 不能代表整个 RAG 系统的真实分布
- **真正的"系统性"判断必须在 Day 10 构造 30+ 条代表性测试集后才能下**

**观察到的方向 (推翻方向高度一致)**:
推翻前分布:    推翻后分布:
L0: 1 (U4)     L0: 0
L1: 0          L1: 0
L2: 0          L2: 3 (U2/U3/U4)
L4: 3 (U1-U3)  L4: 1 (U1)


3 个推翻**全部落到 L2**, 暗示可能存在系统性偏差. 但**这只是暗示, 不是结论**.

**可能的偏差原因 (基于 4 样例的初步推测, 待大样本验证)**:
1. **LLM 自述偏向"拒答"**: LLM 面对模糊情况倾向说"未提及", 这种表述最容易
   被工程师解读为"数据没有"(L4), 而不是"LLM 没搞清楚"(L2)
2. **工程师倾向"上游归因"**: 遇到问题, 人本能地往链路上游找原因 
   (数据 > chunker > 检索 > LLM). L4 在最上游, 最容易被选中
3. **L2 问题反思要求更高**: 判 L2 意味着"我的 LLM/Prompt 不行", 心理上比
   判 L4 ("数据不行") 更有阻力

**初步启示 (有待验证)**: **RAG 项目遇到 LLM 拒答时, 应优先假设 L2 而非 L4,
要主动证伪才能归 L4**. 这是对 Day 4-7 默认做法的一个调整方向.

**Day 10 必做**: 构造代表性测试集 (按公司 / 行业 / 查询类型 / 难度 4 维度
采样, 目标 30-50 条), 在大样本上跑 Day 8 的 3 步法审计, 验证 L2 偏差是不
是真的全局现象.

**面试金句 (修订版, 不夸大)**:
> "我这个项目做到 Day 7 的时候, 几个翻车案例的诊断都是'年报没数据'. 做完
> Day 8 审计才发现, 在 4 个翻车样例上 3 个其实是 LLM 没用好候选信息, 不
> 是数据没有. 这是个**初步发现, 需要更大样本验证**, 但已经让我把 Day 10
> 的优化重心从'修数据源'调整到'先验证再修 Prompt', 方向更谨慎了."

### Q1: 你这个项目里最重要的方法论认知升级是什么?

**可能的追问**:

### Q3: 你项目里改过自己的结论吗? 讲一个具体例子.

5. 长期学到了什么

**完整话术**:

> "Day 7 我做了一次错判, 且整段错判过程有完整的追溯记录.
>
> 当时招商银行不良贷款率这个查询, LLM 答不出完整答案. 我注意到 p.45 chunk
> 里只写 '报告期末不良贷款率 0.93%', 没有 '2025 年' 字样. 我的推理是:
> RecursiveCharacterTextSplitter 按字符切块, 把 '报告期 = 2025 年' 这个
> 上下文切走了, 所以 LLM 不知道是哪年的数据. 我把这个归为 L0 chunking 边界
> 问题, 登记为 TD-7-1.

> Day 8 我跑了一个验证脚本, 把 Top-20 完整 content 全打印出来. 发现两个事:
> 第一, Top-2 (p.5) 明确写了 '2025 年度报告' + '2025 年是很不平凡的一年' —
> LLM 从 Top-5 其他 chunk 就能推断时间, 不需要 p.45 自己带年份; 第二, Top-5
> 里有 3 个不同口径的不良贷款率 (本公司 0.93%、本集团 0.94%、消费信贷 1.52%).
>
> 真正让 LLM 卡住的不是不知道年份, 是不知道用户要哪个口径. 我把这个错误的
> 推理保留在笔记里, 标注 'TD-7-1 归因错误, 真实归因见 TD-8-2', 而不是删掉
> 装作没发生.
>
 还有一点要补: **Day 8 这个修正本身也是基于 4 样例**. 我现在保持开放态度,
> 等 Day 10 大样本评估集结论. 工程师不只要会推翻自己的旧结论, 也要承认新
> 结论可能也需要更多证据支撑.
> 长期学到的是: **我的归因可能是错的, 即使当时看起来很合理**. 所以 Day 8
> 之后我强制自己, 所有关键归因必须有脚本证据, 不能只有推理. 这个教训对我
> 后面做任何 LLM 应用都有价值."

**答题节奏**: 100 秒 (比前两题稍长, 因为要讲清具体案例)
**可能的追问**:

## 6. 已知问题 & 技术债重估

这一段是 Day 8 审计后对技术债的正式重建. 为什么叫"重估"不叫"新增":
因为 Day 8 不只是加了新技术债, **更重要的是重新审视 Day 7 的已有技术债,
把归因错的改正, 把被低估的升级, 把被高估的降级**.

### 6.1 Day 7 技术债的重估

Day 7 一共登记了 4 个技术债 (TD-7-1 到 TD-7-4). Day 8 审计后的状态:

#### TD-7-1 (chunking 边界丢年份锚点) — **🔥 归因错误, 归档**

**Day 7 原描述**:
> "招行 p.45 的 chunk 里只写'报告期末不良贷款率 0.93%', 没有'2025 年'字样.
> RecursiveCharacterTextSplitter 按字符长度切块, 把'报告期 = 2025 年'的
> 上下文切到了其他 chunk. 这是字符级 chunker 的结构性副作用, 影响所有
> 'XX 公司 XX 年 XX 指标'这类查询的召回质量."

**Day 7 原优先级**: P1
**Day 7 原方案**: Day 10 metadata 注入 Prompt (低成本) / Day 11+ 语义 chunking (根治)

**Day 8 审计结论**: **归因错误**
- 脚本 25b 证明 p.45 chunk 本身是 "自足 chunk" (含"报告期末"锚点 + "0.93%"数据)
- 在年报上下文里, "报告期末" 就是足够的时间锚点, 不需要 "2025 年" 字样
- 招行 Top-5 里的其他 chunk (如 p.5 董事长致辞) 明确写了 "2025 年度", LLM
  能从上下文推断时间主体
- **招行拒答的真实原因是"多口径错位"(0.93%/0.94%/1.52% 三个数字 LLM
  不知道选哪个), 不是"年份锚点缺失"**

**处理方式**: **归档, 不删**
- 保留 TD-7-1 作为历史记录, 但明确标记 "归因错误"
- 新归因指向 TD-8-2 (多口径错位)
- Day 10 不做 "为 chunking 边界注入年份" 这类修复

**为什么归档不删**: 认知过程本身是面试的核心资产.
"我 Day 7 以为是 chunker 的问题, Day 8 审计发现是 LLM 作答的问题" — 这
个故事在面试里比 "我从来没判错过" 强 100 倍.

---

#### TD-7-2 (Reranker 延迟不稳定, chunk 长度是主因) — **P2 维持**

**Day 7 原描述**: Reranker 精排耗时 4.5s ~ 15s, 长表格 chunk 填满
max_length=512 是主因.

**Day 8 审计**: 未做专项审计.
Day 8 脚本 27 (参数审计) 原计划测 `max_length` 对延迟的影响, 推迟到 Day 9.

**维持 P2 原因**:
- 延迟问题不影响答案质量
- 可以通过 Reranker 条件开启 (TD-7-4) 间接缓解
- Day 9 审计后再决定具体修法

---

#### TD-7-3 (Reranker 不做精确口径判别) — **P2 → P1 大幅升级**

**Day 7 原描述**:
> "A (整体 0.93%) vs B (房地产业 4.64%) 分数 0.998 vs 0.993 — Cross-Encoder
> 判'都相关', 不判'是不是同一口径'. 口径错位的页面会同排 Top, 依赖 LLM
> 读内容区分."

**Day 7 原优先级**: P2
**Day 7 原判断**: "这是 Cross-Encoder 的能力边界, 不能直接修, 需要 LLM-rerank
架构升级才能解决."

**Day 8 审计后升级为 P1 的理由**:

U4 招行案例完美印证了 TD-7-3: Top-5 有 3 个不同口径的不良贷款率 (本公司
0.93% / 集团 0.94% / 消费信贷 1.52%), LLM 面对这 3 个数字**无法选择**,
输出含糊答案或拒答.

**Day 7 的错在于判定优先级时用的是"能不能直接修"**, 而不是"问题严不严重".
P1 的定义应该是"对系统答案质量影响大", 不是"修起来难不难".

- 影响范围: Day 8 审计 4 个案例里 U4 直接命中, U3 也有部分影响 (Top-5 里子
  公司/母公司口径混杂)
- 严重程度: 多口径错位导致 L2a 拒答, 是这次 75% 推翻率里的主要贡献因素之一

**Day 10 修法** (从"不可修"改为"Prompt 工程可缓解"):
不需要等 LLM-rerank 架构, **在 Prompt 里加口径引导规则就能缓解**:
> "如遇多个口径 (本公司/本集团/业务线), 优先选择与问题字面最匹配的口径,
> 同时列出其他口径供参考. 不要因多口径而拒答."

这是 Day 7 没想到的低成本方案 — **Day 7 过早地把 TD-7-3 归为"架构问题",
忽略了 Prompt 工程这个中间层**.

---

#### TD-7-4 (默认开 Reranker 延迟不可接受, 需条件开启) — **P3 → P2 升级**

**Day 7 原描述**: Reranker 默认开启让所有查询加 5~15s 延迟, 但 9 组中只
有 2 组答案有提升 — 收益稀薄.

**Day 8 审计后升级理由**:
脚本 25b 证明 L2b (排序问题) 是真实存在的 RAG 瓶颈 (U3 就是典型). Reranker
条件开启不只是"延迟优化", 也是**"L2b 修复的核心手段"**. 从纯延迟考虑升级
为兼顾延迟和答案质量.

**Day 10 修法**:
扩展 Query Router, 加一条规则:
- 查询包含 "具体数字" 类问题 (利润/营收/比率等) → 开 Reranker
- 查询包含 "综合/概况/情况" 类抽象词 → 不开 (L2c 语义粒度错配问题, Reranker 无用)
- 查询包含 "时间/年份/季度" → 根据后缀名词决定

### 6.2 Day 8 新增技术债

#### TD-8-1 (表格切断 chunk) — **P3 个案级 (基于 1/5 形态量化, 待 TD-8-4 全量后重审)**

**描述**:
利润表/收入表等财务数据表的"表头行"(含"单位:元""币种:人民币"等标识)
被 chunker 切分成独立 chunk, 数字部分在下一个 chunk. 检索时"表头 chunk"
因含财务关键词被召回到 Top-5, 数字 chunk 没进 Top-20, 导致 LLM 只看到
表头不到数字.

**⚠️ 命名修正记录 (Day 8 收工时)**:
- 本 TD 早期叫"碎片 chunk", 后改为"表格切断 chunk", 因为"碎片"在中文里
  语义过于宽泛
- 脚本文件名同步修正: `26_audit_chunking_fragment.py` → `26_audit_table_fragment.py`

**⚠️ 量化范围限制**:
TD-8-1 只覆盖 chunker 切断的 1 种形态 (表格切断). 其他 4 种切断形态登记
为 TD-8-4 由 Day 9 量化:
- 指代切断 ("本公司"无公司名)
- 时间锚点切断 ("报告期末"无年份)  
- 口径说明切断 (数字和口径定义分离)
- 叙事切断 ("因此..." 但原因在前一 chunk)

**P3 判定的前提**: **仅在"表格切断"这 1/5 形态上的全局比例 4.5%, 判 P3
个案级**. 如果 TD-8-4 全量量化后发现其他形态合计比例高 (如总和 >20%),
**TD-8-1 + TD-8-4 应作为整体重新评估优先级**.

**证据 / 量化** (仅表格切断形态):

  占含财务关键词 chunks 的 4.5%)
- 按公司分布: 宁德时代 22.6% (最高), 比亚迪 9.5%, 招行 0% (最低)

**修复方案 (仅针对表格切断)**:
- **优先**: Day 10 Prompt 工程时, 对"财务数字"类查询在 Prompt 里加引导
- **次选**: Day 11+ 如果评估集发现 L2b 命中率高, 做表格感知的 chunker
- **不做**: 专项修复 chunker (投入产出比太低)

#### TD-8-2 (LLM 多口径错位) — **P1 核心 Day 10 主战场**

**描述**:
当 Top-5 包含多个不同口径的数据 (本公司 / 本集团 / 子公司 / 业务线) 时,
LLM 无法判断用户想要哪个口径, 选择拒答或给出含糊答案.

**本质**: 这是 TD-7-3 的真实表现, Day 7 低估其影响, Day 8 升级.

**证据**:
- U4 招行: Top-5 有 3 个不良贷款率 (0.93% / 0.94% / 1.52%), LLM 拒答

**影响范围**:
