## 3. 核心知识点

### 3.1 D7: 实验设计错误 > 代码 bug (Task 1 催生)

**现象**: Task 1 报告显示 "top_k 扫描命中率从 4.7% → 26.1% 单调上升",
表面看 "top_k=15 最优". 但与预估 "top_k=5 最优或微调到 7" **严重不符**.

**根因**: `hybrid.py:157`
```python
recall_k = top_k * self.recall_multiplier
```

top_k **不是独立参数**, 它通过 `recall_k` 同时决定两路候选池大小.
扫 top_k 实际上在扫 `recall_k`, 没法分离影响.

**教训**: 读过 hybrid.py 的我, 自我攻击清单只检查了 "RRF k 独立性" (FP2),
**漏掉 top_k 和 recall_mult 的耦合**. 这是工程师和研究员的分水岭 — 写完规则前**必须列参数间耦合检查表**.

**新规则 (自我攻击清单第 6 步)**:
> 每次设计参数扫描前, 强制列耦合表:
> 1. 本参数是否依赖其他参数? (看源码里本参数的所有使用位置)
> 2. 其他参数是否通过本参数间接影响结果? (看复合公式)
> 3. 扫描的是真正的独立维度吗? 如果不是, 解耦再扫

### 3.3 D9: 评估函数设计陷阱 (Task 2 假阳性催生)

**现象**: Task 2 Phase 2 报告显示 U5 紫金 **10/10 全部答对**, 与 Task 1 的
"U5 是死穴" 严重矛盾.

**根因**: `check_answer()` 对 U5 用 OR 逻辑:
```python
if "地缘政治" in answer and ("海外" in answer) and "风险" in answer:
    return True
```

**LLM 拒答文本会复述关键词**:
> "根据已有资料无法回答此问题. 参考资料中提到了紫金的**海外业务**和
> **地缘政治风险**的应对措施, 但未提供海外业务具体的营收数据."

拒答态触发关键词, 判成答对. **100% 假阳性**.

**新规则 (评估函数设计自我攻击清单 FP7)**:
> 每次设计"判答对"函数前, 列 3 个场景:
> 1. 标准答对: 含目标数字 + 明确陈述 → True
> 2. 标准答错: 不含关键信息 + 说"无法回答" → False
> 3. **⚠️ 拒答态**: 说"无法回答"但复述关键词 → 假阳性陷阱
>
> 修法: 先检测拒答语气, 拒答态即使含所有关键词也判 False
> ```python
> REFUSAL_PATTERNS = ["根据已有资料无法回答", "无法回答", "未提供", ...]
> ```

### 3.4 RRF 等权融合对金融场景的局限

**发现**: U5 BM25 Top-20 命中 7/9 (Rank 1 就是金页), 但 Hybrid Top-5 变 **0/9**.

**数学解释**:

RRF 公式: `score = 1/(k + rank_dense) + 1/(k + rank_bm25)`, k=60.

- 场景 A: BM25 Rank 1, Dense 不命中 → score = 1/61 + 0 = **0.0164**
- 场景 B: Dense Rank 3, BM25 Rank 80 → score = 1/63 + 1/140 = **0.0230**

**场景 B 的 RRF 分反而更高**, 但它不一定是金页.

**含义**: 当两路能力差异大时 (BM25 >> Dense), 等权 RRF 会 **稀释 BM25 的强信号**.
金融数字类查询 BM25 普遍强于 Dense, 因此 RRF 等权在这个场景下有系统性局限.

**登记为 TD-9-3 候选 (P3)**, Day 12 RAGAS 评估后决定是否升级成正式 TD.

### 3.5 LLM 对数据噪声的韧性 (Task 2 最意外收获)

**U6 证据**:
- 10 次调用所有 mode 下 Top-5 **金页 0/6 命中**
- 但 LLM **10/10 答对精确数字** "423,701,834 千元"
- Top-1 稳定是 p.108 (审计报告复述页)

**含义**: LLM 不需要完美金页. 只要召回 chunk **包含目标数字** (不管来源页),
LLM 就能答对. 这和 Day 7 "Reranker 能力边界" 是反面 — **LLM 对数据噪声的容忍度超预期**.

**对评估的冲击**: "Top-K 金页命中率" 作为 RAG 评估指标有**根本局限** —
低命中率 ≠ 答不对.

### 3.6 L2a 多口径错位: 从特例到通用模式

**Day 8 定案**: U4 招行 0.93% 本公司 / 0.94% 集团 / 1.52% 消费信贷, LLM 不知选哪个 (L2a 特例).

**Day 9 扩展**: U3 国电南自 "召回的子公司净利润 427M, 但查询的是集团合并口径".
LLM **正确拒答**, 明确说 "参考资料提供子公司数据但未提供集团合并".

这是 L2a 的**新变体**:
- U4 型: 同一报告内有多口径并列, LLM 不知选哪个
- U3 型: 召回的口径和查询不匹配, LLM 应当明确拒答

## 4. 关键实验发现 (5 个故事)

### 故事 1: Task 1 参数扫描数据反常 → 发现 top_k 和 recall_mult 耦合 (D7)

**场景**: 跑完 84 次检索的参数扫描报告, 看到 top_k 扫描 **单调上升 4.7% → 26.1%**.

**第一直觉**: "top_k=15 最优". 但这与我的预估 "top_k=5 维持或微调 7" 严重不符.
按自我攻击清单第 5 条硬规则 (预估偏差 2 倍以上要警觉), 启动排查.

**诊断 (debug_topk.py 独立验证)**:
```
top_k=3 : Top-3 页号 = [34, 5, 35]                      ← 和其他组完全不同
top_k=5 : Top-5 页号 = [45, 5, 50, 31, 34]              ← 前缀关系成立
top_k=10: Top-10 页号 = [45, 5, 50, 34, 31, 36, 15, ...] 
```

top_k=5 vs 10 vs 15 是前缀关系 ✅, 但 **top_k=3 完全独立**.

**根因**: `hybrid.py:157` `recall_k = top_k × recall_multiplier`. 扫 top_k 实际是扫 recall_k.

**修正结论**: recall_mult=4 是真结论 (逐查询显示拐点在 4), 其他参数维持默认.

**工程价值**: 这不是代码 bug, 是**实验设计错误**. 代码运行完全正确, 数据也是真的,
但变量没解耦导致结论无意义. **工程师最大的风险不是解决不了问题, 是用错方法诊断问题**.

### 故事 2: Phase 1 发现 BM25 >> Dense 但 RRF 反而稀释 (技术洞见)

**场景**: Task 2 Phase 1 读 3way cache 做三路分工分析, 看到 U5 紫金:
```
BM25 Top-20: 命中 7/9, 首位 Rank 1   ← BM25 大胜
Hybrid Top-5: 命中 0/9                ← RRF 后全没
```

**诊断 (RRF 数学)**: 两路 RRF 等权融合. 对于 "BM25 Rank 1 + Dense 不命中" 的文档,
RRF 分只有 1/61 = 0.0164. 而 "Dense Rank 3 + BM25 Rank 80" 的文档 RRF 分 0.0230.
**Dense 哪怕弱命中也能压过 BM25 强命中**.

**启示**: 金融数字类查询本质上是 BM25-dominant (精确术语匹配强, 语义匹配弱).
RRF 等权融合不适合这种场景, 可能需要**加权 RRF (BM25 权重 > Dense)**.

**面试价值**: 这个发现展示两个深度:
- 对 RAG 技术栈的**数学级理解** (不是盲用教科书 RRF 公式)
- 对业务场景的**敏感度** (金融 vs 开放域语义需要不同融合策略)

### 故事 3: Task 2 Phase 2 假阳性 → check_answer 拒答态 bug (D9)

**场景**: Phase 2 跑完 30 次, 报告显示 U5 紫金 **10/10 全对**, 但我记得 U5 是 Day 6-7 死穴.
按偏好 5 我没有立即相信这个"好消息", 让用户从 JSON 挑 4 条 answer_preview 看原文.

**真相**:
```
U5 典型答案: "根据已有资料无法回答此问题. 参考资料中提到了紫金的**海外业务**布局
和**地缘政治风险**的应对措施, 但未提供海外业务具体的营收数据."
```

LLM 实际在**拒答**, 但答案复述了查询关键词 "海外+地缘政治+风险", 触发了我写的 OR 判定.

**修正方式**: 先检测拒答语气 (优先级最高), 拒答态即使含所有关键词也判 False.

**教训的两层**:
- 表层: 评估函数要加 REFUSAL_PATTERNS 预过滤
- **深层**: 评估指标本身需要自我质疑. "看起来结果太好 vs 已知事实" 的矛盾是关键信号

**面试价值**: 展示 **"发现自己的 bug 比发现别人的 bug 更难"** 的元认知.
这类错误不是代码 bug, 是**评估设计 bug**, 和 D7 的"实验设计 bug" 同类.

### 故事 4: U6 金页 0/6 但 LLM 10/10 答对 (审计报告复述页)

**场景**: U6 宁德 2025 营业收入, 金页集是 p.11/24/116/118/195/231 (主披露页).
10 次调用 **全部 mode Top-5 都没进金页**, 但答案全部精确到位.

**真相**: Top-1 稳定是 **p.108 (审计报告复述页)**, 页面内容:
> "宁德时代公司 2025年度实现收入 423,701,834 千元, 较上年增长 17.04%..."

**我之前在金页标注时把 p.108 标为 false** (认为审计报告是"复述"不是"披露"). 但实际:
- 审计报告是**合规披露的组成部分** (第八节 "财务报告" 下的"审计报告")
- 对 LLM 来说, **任何含精确数字的 chunk 都是有效答案源**
- "主披露页 vs 复述页" 是我强加的分类, RAG 系统没这个概念

**三重含义**:
1. **金页标注系统性偏严** → TD-9-1b 第三次印证, Day 11 RAGAS 评估前要重新定义 ground truth
2. **LLM 韧性超预期** → 不需要完美金页, chunk 含目标数字就够
3. **Top-K 金页命中率有根本局限** 作为评估指标 → RAG 评估必须看答案质量, 不是中间步骤的 proxy

**面试价值**: 这个故事可以驳斥 "RAG 就是做好向量检索" 的肤浅理解.
**评估指标本身的有效性, 比指标数值更关键**.

### 故事 5: Task 2 设计反复翻车 → project_map.md 诞生 (D8)

**场景**: Task 2 设计 3 次, 前 2 次都因为我**错估项目现状**翻车:

## 7. 自我检验 (10 题 3 层次)

### 基础事实 (3 题)

**Q1**: Day 9 最终确定的检索层参数是什么? 每个参数的扫描结论是什么?

A: top_k=5 (扫描无效, 非独立参数), rrf_k=60 (6 样本下不敏感维持默认),
recall_multiplier=4 (真最优, 拐点在 4, ≥6 时 U5 紫金命中率从 11% 掉到 0%).

**Q2**: U5 紫金 "海外业务营收和风险" 查询, BM25 Top-20 和 Hybrid Top-5 的金页命中数分别是多少?

A: BM25 Top-20 命中 7/9 (Rank 1 就是金页), Hybrid Top-5 命中 0/9.
这是 RRF 等权融合稀释 BM25 强信号的证据.

**Q3**: U6 宁德 "2025 年营业收入" 查询, 10 次 AdvancedRAG 调用中, LLM 答对了几次? 为什么?

A: 10/10 全部答对. 因为 Top-1 稳定是 p.108 (审计报告复述页), 含精确数字 "423,701,834 千元".
尽管 p.108 不在我标注的金页集 {11, 24, 116, 118, 195, 231} 里, 但对 LLM 来说它是有效答案源.

### 诊断方法 (4 题)

**Q4**: Task 1 报告显示 top_k 扫描命中率单调上升, 你怎么判断这不是真结论?

A: 3 步.
1. 对比预估: 我的预估是 "top_k=5 维持或微调 7", 实际 "top_k=15 最优 (26.1%)" 严重不符 (自我攻击清单第 5 条)
2. 写 30 行独立验证脚本 debug_topk.py, 用固定参数跑不同 top_k, 对比 Top-K 列表
3. 观察到 top_k=5/10/15 是前缀关系, top_k=3 独立, 读源码发现 `recall_k = top_k × recall_multiplier`

教训: **扫复合参数无效**, 必须解耦扫真正的独立维度.

**Q5**: U5 报告"10/10 全对" 你是怎么发现是假阳性的?

A: 我没有立即相信 "U5 全对" 这个好消息, 因为它和 Task 1 "U5 是死穴" 的已知事实矛盾.
让用户挑 4 条 answer_preview 看原文, 发现 U5 典型答案是 **"根据已有资料无法回答"**,
但复述了 "海外+地缘政治+风险" 关键词, 触发了我写的 OR 判定 — **拒答态假阳性陷阱**.

**Q6**: 怎么区分 L1 (召回失败) / L2a (多口径错位) / L2b (排序问题)?

A:
- L1: Dense Top-20 + BM25 Top-20 都没召回金页
- L2a: 召回了但 LLM 在多个候选间选错口径, 或召回口径和查询不匹配时未明确拒答
- L2b: 金页召回了但排名靠后 (比如 Rank 15/18), 没进 Top-5, LLM 看不到

Task 2 数据:
- U3: BM25 召回了 1/9, Hybrid Top-5 0/9, LLM 拒答明确说子公司/集团口径 → L2a
- U5: BM25 召回 7/9 但 Hybrid Top-5 0/9, RRF 稀释 → 介于 L1 和 L2b 之间, 实际是 RRF 融合问题
- U6: Top-5 0/6 但 LLM 用 non-gold 页答对 → 检索指标失效, 需重新看

**Q7**: project_map.md 为什么要提前到 Day 9 建, 不等 Day 10 开工前?

Task 2 设计连续翻车 2 次都因为我错估项目现状, 这是 "不建就会继续翻车" 的触发点.

### 设计判断 (3 题)

**Q8**: Day 10 的 Prompt 工程要处理哪几类多口径错位场景? 规则怎么写?

A: 2 类场景.

场景 1 (U4 型: 同报告内多口径并列):
> Prompt 规则: "如果参考资料中出现多个数字口径 (如本公司/集团/子公司),
> 优先选集团合并口径, 并明确标注使用的口径."

场景 2 (U3 型: 召回口径和查询不匹配):
> Prompt 规则: "如果用户查询是集团合并数据, 但参考资料只有子公司或子板块数据,
> 明确回答 '已召回 X 数据, 但集团合并口径未在参考资料中找到', 不要用子口径代答."

注意: U3 的 LLM 已经做对了 (正确拒答), Prompt 规则是**强化现有正确行为**, 不是修复错误.

**Q9**: 如果 Day 10 发现 Prompt 工程也救不了 U3/U4, 你的下一步思路是什么?

A: 分级回退.
1. 首先检查 Prompt 规则是否真的生效 (打印实际发给 LLM 的 system prompt)
2. 如果生效但 LLM 忽视规则, 换更强 LLM (DeepSeek → GPT-4)
3. 如果换模型也没救, 考虑 **L3 级修法**: LLM 作答前先用轻量 Agent 对多口径 chunks 做 "口径归一化预处理"
4. 最后底线: 在前端标注 "多口径场景, 答案可能不完整", 交给用户判断

**但最可能的情况是 Prompt 工程就够了**, 因为 U3 的 LLM 已经能正确拒答,
U4 的 LLM 是在多口径并列时没明确归一化口径, 这类问题 Prompt 工程的历史成功率很高.

**Q10**: Day 9 做了 3 次 Task 2 设计, 每次翻车的根因是什么? 怎么防止 Day 10 再翻车?

A: 3 次翻车根因分别是:
1. 第 1 次: 不知道 Day 6 已做 Multi-Query → **没盘点项目已有能力**
2. 第 2 次: 不知道 Day 7 AdvancedRAG 已集成 Reranker → **错读了源码快照**  
3. 第 3 次 (假阳性): 评估函数设计没考虑拒答态 → **没穷举评估场景**

Day 10 防御:
- project_map.md 第一件事就读, 确认项目现状

- 评估函数设计含拒答态场景 (FP7)
- **每次发现翻车, 登记新的自我攻击清单条目** — 这是最重要的机制. 目前清单已 7 条 (FP1-FP7), 每条都是血的教训.
