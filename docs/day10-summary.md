### 2.2 U4 (招商银行 2025 不良贷款率) 三轮对比

U4 是 TD-8-2 经典样本, Day 8 审计显示 Top-5 应有 3 个口径 (本公司 0.93% /
集团 0.94% / 消费信贷 1.52%), Day 8 纯 Prompt v1 下 LLM 直接拒答.

| 版本 | `is_refusal` | 列出口径数 | 答案实质 |
|---|---|---|---|
| v1 (Day 8 基线) | True | 0 | 全文拒答 |
| v2 | True | 2 | 集团 0.94% + 消费信贷 1.52%, 明说缺本公司整体 |
| **v3** | True (pattern 末尾命中) | **3** | 集团余额 68.2B + 本行余额 64.0B + 消费信贷 1.52%, 说明缺贷款总额所以算不出率 |
| v4 | False | 1 | "招商银行2025年不良贷款率为0.94% [5]。" 一句话 |

**关键发现**: v3 信息量最丰富 (3 口径), v4 尽管 `is_refusal=False` 但**信息量反而下降**
到 1 口径. 评估指标和实际质量脱钩.

### 2.3 U3 (国电南自 2024 归母净利润) 三轮对比

U3 是 L2a 扩展变体 (子公司/集团错位), Day 9 Phase 2 即能部分作答.

| 版本 | `is_refusal` | 答案开头 | 答案实质 |
|---|---|---|---|
| v1 (Day 9 基线) | True | "根据已有资料无法回答此问题" | 子公司 427,419,723 + 115,172,643, 说明缺集团 |
| v2 (T2, hyde mode) | True | "根据已有资料无法回答此问题" | 只提"5.82亿元但不明确是否归母" (hyde 召回漂走, 没命中 p.251 子公司页) |
| v2 (hybrid mode 诊断) | True | "根据已有资料无法回答此问题" | 子公司 427M + 说明缺集团 (召回到 p.251) |
| v3 | True | "根据已有资料无法回答此问题" | 只提"5.82亿元但不明确是否归母" (hyde 召回再次漂走) |
| v4 | True | "根据已有资料无法回答此问题" | 只提"5.82亿元" |

都照字面理解但选择忽略, **开头的拒答句一直存在**. 这是 TD-10-1 的直接证据.

### 2.4 Top-5 召回稳定性数据

同一 query (U3 国电南自净利润), 不同 mode 下 Top-5 召回**完全不同**:

| mode | Top-5 页号 (国电南自) | 包含 p.251 (子公司数字页) |
|---|---|---|
| v2 auto → hyde | 32 / 126 / 101 / 80 / 54 | ❌ |
| v2 hybrid 诊断 | 213 / 251 / 32 / 282 / 212 | ✅ |
| v3 auto → hyde | 11 / 32 / 126 / 101 / 7 | ❌ |
| v4 auto → hyde | 11 / 32 / 126 / 101 / 7 | ❌ |

**关键发现**: auto → hyde 模式召回稳定 (v3 v4 两次 Top-5 相同), 但**和 hybrid 模式

## 3. 核心知识点 — 指令工程的三个边界

Day 10 三轮迭代揭示了大模型指令遵循的三个边界. 这三个边界是从实验数据里
逆向归纳出来的, 不是先验理论.

### 3.1 否定指令遵循弱于肯定指令

**现象**:
- v3 规则 6 含"**不要**在开头添加'根据已有资料无法回答'" — LLM 依然添加
- v4 规则 6 含"**必须以** '已找到 X' **开头**, **不得以** '无法回答' 开头" —
  LLM 依然开头写"根据已有资料无法回答"

**机制猜想** (假设, 未严格证伪):
大模型训练时, "不要做 X" 类指令需要模型先理解 X, 再产生对 X 的抑制. 而
训练数据里"拒答"是一个强先验 (RLHF 为了安全性鼓励 LLM 在不确定时拒答).
**否定指令的抑制强度 < 拒答先验的吸引力**, 所以 LLM "字面理解但不遵守".

**修法不是"加更强的否定"**, 因为 v4 比 v3 更强也没用. 可能的修法方向:
- 结构强制 (JSON schema / function calling) — 不是 Prompt 工程层能解的
- 换 LLM (其他模型可能先验不同)
- 训练层 (自己微调一个 RAG-friendly 模型)

**登记**: TD-10-1 "DeepSeek 否定指令遵循度弱". Day 11+ 做 RAGAS 评估时, 看
这类场景在大样本上的比例, 再决定是不是值得结构化改造.

### 3.2 规则复杂度和 LLM 遵循度之间有拐点 (U4 现象)

**现象**:
- v2 (7 条规则, 402 字): U4 列 2 口径
- v3 (7 条规则, 472 字): U4 列 **3 口径** ← 拐点最优
- v4 (7 条规则, 491 字): U4 只列 1 口径 ← **过度约束退化**

v3 → v4 仅加了 19 个字 (规则 6 的"必须以开头...不得以开头"), 结果 LLM 把
U4 答案从 3 口径塌缩到 1 口径. 这 19 字**不是增加了信息, 是增加了约束**.

**机制猜想**:
当 Prompt 规则足够复杂时, LLM 无法同时满足所有规则, 会选择"**最保守的最小
满足策略**": 给一个最能不触发任何规则的简短答案, 就 0.94% 一个数字. 比
"列 3 口径但可能违反某条细则" 对 LLM 而言是更安全的选择.

**工程含义**:
**Prompt 工程不是线性优化**. 规则越多越细不等于 LLM 越听话. 存在一个 U
型曲线, 拐点在某个中等复杂度. 找拐点**必须靠实验, 不能靠直觉**.

### 3.3 评估指标和答案质量脱钩 (is_refusal FN-2)

**现象**:
- v3 U4: `is_refusal=True`, 但答案列了 3 个口径有引用有数值 (答案实质"好")
- v4 U4: `is_refusal=False`, 但答案只一句话 0.94% (答案实质"差")

如果只看 `is_refusal`, **v4 比 v3 好**. 但如果肉眼看答案, **v3 远好于 v4**.
自动化评估指标把错误版本评成了胜者.

**根因**: `is_refusal` 只检测 8 条 REFUSAL_PATTERNS 是否出现, 不看上下文
位置、不看除拒答句外还说了什么. 开头拒答 + 列 3 口径, pattern 命中即判拒答,
但用户读到的是信息量高的半作答.

**这是 Day 9 D9 元教训的延伸**: D9 教会"先判拒答再判关键词", 但没教"拒答
之后还说了啥". FN-2 是 Day 10 新发现, 升级为:

> **评估函数不仅要检测拒答态, 还要检测"拒答之后是否还有实质内容"**.
> 拒答 + 半作答 ≠ 纯拒答, 评估要能区分.

**登记**: TD-10-3 "is_refusal FN-2 需升级". 修法方向: 检测拒答 pattern 后
剩余字符数 / 含数字密度 / 含引用标 `[N]` 等. Day 11 RAGAS 前修.

## 4. 关键实验发现 — v2 → v3 → v4 三段故事

### 4.1 v2: 规则 5/6 初次落地, 发现规则 2/6 打架

**设计**:
v1 (5 条规则) 扩到 v2 (7 条规则), 新增:
- 规则 5: 多口径并列时优先字面匹配口径, 列其他口径供参考, 不要因此拒答
- 规则 6: 口径不匹配时, 明确告知"已找到 X, 但未找到 Y"

**T2 跑完发现**:
- U4 朝正向动了一步: 从全文拒答 → 列 2 口径 (集团 0.94% + 消费信贷 1.52%).
  但 **0.93% 本公司口径没出现** (召回没命中那个 chunk)
- U3 **回退**了: hyde 模式下召回漂走没命中 p.251 子公司页, LLM 只拿到
  "5.82亿" 一个数字, 开头还是"根据已有资料无法回答"

**诊断 U3 回退**:
为了排除是 "hyde 随机性" 还是 "Prompt 副作用", 用 hybrid 模式再跑 U3 一次:
mode=hybrid → Top-5 含 p.251 ✅ → 答案 "子公司 427M... 但未提供集团..."
但开头仍是"根据已有资料无法回答"

证据说明: 召回没问题, **问题是规则 2 和规则 6 在 U3 场景打架**.
- 规则 2: "资料不足就拒答"
- 规则 6: "口径不匹配就说 '已找到 X 但未找到 Y'"
U3 场景规则 6 应该赢, 但 LLM 两条都照做, 变成"拒答句 + 补充子公司数据".

**登记 TD-10-5 候选**: 多规则打架的优先级机制. v3 起手修.

### 4.2 v3: 规则 2 收紧 + 规则 6 加否定指令, U4 进一步好转, U3 未救

**设计**:
- 规则 2: "没有足够信息" → "**完全不涉及问题主题**" (触发门槛提高)
- 规则 6: 加一句 "不要在开头添加'根据已有资料无法回答'" (否定指令)

**T2 跑完发现**:
- U4 **继续朝正向动**: 从 v2 的 2 口径 → v3 的 **3 口径** (集团余额 68.2B +
  本行余额 64.0B + 消费信贷 1.52%), 说明缺贷款总额所以算不出率. 信息量最大
- U3 **仍未救回**: 开头还是 "根据已有资料无法回答", 和 v2 一样. 规则 6 的
  否定指令完全失效

**判断**: v3 是三轮 Prompt 的**局部最优**. U3 的问题**不是 Prompt 层能解**,
是 DeepSeek 否定指令遵循度的天花板.

### 4.3 v4: 强化否定为肯定 + 强制句式, 过度约束退化

**设计假设**:
如果 DeepSeek 对否定指令弱, 那把 "不要以 X 开头" 改成 "**必须以** Y 开头"
这种肯定指令, 可能有效. 规则 6 改为:
> 回答必须以"已找到 X 口径的数据 [引用] 为 <数值>"或"参考资料中包含 X 口径
> 数据"开头, 不得以"根据已有资料无法回答"开头.

**T2 跑完发现**:
- U3: 开头仍是 "根据已有资料无法回答" — 肯定指令**也没用**
- U4: 从 v3 的 3 口径**塌缩到 1 口径** — "招商银行2025年不良贷款率为 0.94% [5]。"
  就一句话

**反直觉**: U4 `is_refusal=False` 但答案信息量**远低于** v3. 如果只看拒答检测,
会把 v4 错判为比 v3 好.

**回滚决策**:
数据支撑 v3 > v4. 最终定格 v3. v4 规则 6 的强化指令**没有正收益, 但有负收益**,
属于典型的"过度 Prompt 工程".

### 4.4 Day 9 U3 基线的认知更正 (D8 元规则延伸)

在查 `day09-l1-diagnosis-results.json` 修 `load_u3_day9_answer` 的 bug 时,
发现一个认知错误.

**我之前的认知**: Day 9 Phase 2 U3 基线是 "部分作答" (基于 `day09-l1-diagnosis.md`
第 4.3 节的引述: "参考资料中提供了国电南自部分子公司...但未提供国电南自
母公司或整个集团合并口径的2024年度净利润总额").

**从原始 JSON 看到的真实基线**:
"answer_preview": "根据已有资料无法回答此问题。\n\n参考资料中提供了国电
南自部分子公司2023年和2024年的净利润数据，以及审计报告确认其财务报表公
允反映了经营成果，但未提供国电南自合并层面2024年度的净利润总额 [2][4]。"


**认知错误定位**: Day 9 U3 **开头第一句就是拒答句**. `day09-l1-diagnosis.md`
引述只截取了后半段 (子公司数据), 我误当成答案开头. 这意味着**v2/v3/v4 U3
"开头拒答" 根本不是 Day 10 的新回退, 是 Day 9 就有的现象**.

**元规则延伸 (D8 在 Day 10 的新注脚)**:
Day 9 建立的 D8 说 "`/mnt/project/` 快照不是权威源". Day 10 发现延伸:
**引述性文档 (如 day09-l1-diagnosis.md) 也不是权威源, 原始数据 JSON / CSV 才是**.
`day09-l1-diagnosis.md` 是对原始数据的解读, 解读会选择性截取.
**元规则升级**: 下游判断前, 优先读原始数据, 引述性文档作辅助.

这个教训登记在 `day09-decisions.md` 的 **D10 元教训** 条目里 (Day 11 前补).

## 5. 面试话术

三个 Day 10 专属问题, 每个 90 秒. 重点是讲**反直觉发现 + 工程克制**,
不讲 "我怎么做 Prompt" 的流水账.

### Q1: 你做过 Prompt 工程吗? 最有意思的发现是什么?

> "Day 10 我做了 3 轮 Prompt 迭代, 试图用 Prompt 工程救回两个多口径错位样本.
> 最反直觉的发现是 **v3 → v4 的退化**: v3 的 Prompt 能让 LLM 列 3 个口径
> (集团余额/本行余额/消费信贷率), v4 我在规则 6 里加强了 19 个字的强制句式
> ('必须以 X 开头, 不得以 Y 开头'), 结果 LLM 答案从 3 口径**塌缩到 1 口径** —
> 一句话 '不良贷款率为 0.94%' 就结束.
>
> 这不是 LLM 变笨了, 是 LLM 面对过度复杂的规则时, 选择了**最保守的最小
> 满足策略** — 给一个简短到不会违反任何规则的答案, 比给一个丰富但可能违反
> 某条细则的答案更'安全'.
>
> **含义**: Prompt 工程不是线性优化. 规则越多越细不等于 LLM 越听话.
> 存在一个 U 型曲线, 拐点在某个中等复杂度. 我最终回滚到 v3 — 这是**数据
> 驱动的工程克制**, 不是'一次成功'的完美故事.
>
> 面试官可能追问: 你怎么知道 v3 是拐点而不是 v5/v6?
> 答: 不知道. 6 样本不足以下系统结论, 这是 Day 11 RAGAS 大样本评估要做的事.
> 今天我选了 '数据支撑最好的版本 + 登记技术债' 而不是 '继续往下雕花'."

**答题结构**: 实验发现 → 反直觉数据 → 机制解释 → **升华到方法论克制**

### Q2: DeepSeek 对指令遵循有什么限制吗?

> "Day 10 我发现一个具体的限制: **否定指令遵循度弱于肯定指令, 但都弱于
> '拒答'这个强先验**.
>
> 具体场景: 多口径场景下, 我在 Prompt 规则 6 里明确写 '**不要**在开头添加
> 根据已有资料无法回答' — 这是否定指令. LLM 照样在开头加. 后来我改成肯定
> 指令 '**必须以** 已找到 X 开头' — LLM 还是开头写拒答句, 不遵循.
>
> 机制猜想: RLHF 训练时 '不确定就拒答' 是一个强安全先验, 被训得特别强.
> Prompt 层的否定/肯定指令都无法压制这个先验.
>
> **修法**: Prompt 层已经到顶, 真要修得靠结构强制 (比如让 LLM 输出 JSON
> schema, 开头字段类型限死) 或者换模型. 我登记为 TD-10-1, 不在 Day 10
> 的范围, 放到 Day 11+ RAGAS 评估有大样本数据后再决定是否值得做结构化改造.
>
> **这类教训比'我搞定了 Prompt' 更有价值** — 因为它划清了 Prompt 工程的
> 能力边界. 工程师应该知道工具的极限在哪儿."

**答题结构**: 现象 → 实验证据 → 机制猜想 → **边界划定 + 工程克制**

### Q3: 你的评估指标有什么局限吗?

> "Day 10 我踩了一个 **自动化评估指标和答案质量脱钩** 的坑.
>
> 背景: Day 9 我写过一个 `is_refusal()` 函数检测 LLM 是否拒答, 用 8 条
> REFUSAL_PATTERNS (如 '根据已有资料无法回答' / '未提供' 等) 做字符串匹配.
>
> Day 10 v4 的 U4 结果: `is_refusal=False` (因为答案里没有任何拒答 pattern).
> 但肉眼看答案只有一句 '不良贷款率为 0.94%'. 同样的 U4 在 v3 下
> `is_refusal=True` (末尾'未提供'触发), 但答案列了 3 个口径有数据有引用.
>
> **如果只看 is_refusal, v4 比 v3 好. 但实质 v3 好 3 倍**.
>
> 根因: `is_refusal` 只看 pattern 是否出现, 不看 pattern 之后还说了什么.
> LLM 的新行为模式 '拒答句 + 半作答拼接' (Day 10 三轮 Prompt 催生) 是我
> 设计 `is_refusal` 时没想到的场景. 这是评估函数的 **FN-2 盲区** (我在
> docstring 里诚实写了这个已知局限).
>
> **含义**: 自动化评估指标必须和肉眼抽查并行. 单一指标很容易被'hack'
> (这次是被 LLM 自己 hack, 不是恶意的, 是 LLM 自然行为). Day 11 上 RAGAS
> 时要多指标交叉 (Faithfulness + Answer Relevancy + 人工抽检).
>
> **升级后的自我攻击清单**: 评估函数不仅要检测拒答态, 还要检测'拒答之后
> 是否还有实质内容'. 这是 Day 10 登记的 FN-8 (or TD-10-3)."

**答题结构**: 场景 + 反直觉指标反转 → 根因 → **多指标交叉 + 工程克制**
