### Phase 1 核心发现

**BM25 在金融数字类查询上全面占优, Dense 几乎全败**:
- U2/U4/U5 BM25 Top-20 命中率 ≥70%
- Dense Top-20 命中率普遍 <25% (U4 是例外, 50%)

**Hybrid RRF 融合反而稀释了 BM25 的强信号**:
- U5 最极端: BM25 Top-1 就是金页 (7/9), RRF 后 Hybrid Top-5 变 0/9
- U2: BM25 Top-1 金页, Hybrid Top-5 只进 1/4
- **RRF 等权融合对"一路强一路弱"场景不友好**

### 为什么 RRF 会稀释强信号

RRF 公式: `score = 1/(k+rank_dense) + 1/(k+rank_bm25)`, k=60.

- 场景 A: 某文档 BM25 Rank 1, Dense 不命中 → score = 1/61 + 0 = 0.0164
- 场景 B: 某文档 Dense Rank 3, BM25 Rank 80 (弱命中) → score = 1/63 + 1/140 = 0.0159 + 0.0071 = 0.0230

**场景 B 的 RRF 分反而更高**, 但它不一定是金页. 这就是 U5 BM25 Rank 1 金页被稀释的机制.

## 3. 假阳性修正说明 (check_answer 函数 bug)

### 问题定位

原 `check_answer()` 对 U5 的判定逻辑:
```python
# U5 特判: OR 逻辑, 任一组合命中即算对
has_geopolitical = "地缘政治" in answer
has_overseas = "海外" in answer
has_risk = any(w in answer for w in ["风险", "应对", "社区", "治理"])
if (has_geopolitical or has_overseas) and has_risk:
    return True, "..."
```

### 实际发生

LLM 的拒答文本 **同时含** "海外" + "地缘政治" + "风险" 这几个词,
因为拒答时 LLM 会**复述查询词 + 描述参考资料里的相关内容**, 然后才说"但未提供具体数据".

示例答案 (U5 hybrid RR-OFF):
> "根据已有资料无法回答此问题. 参考资料中提到了紫金矿业的**海外业务**布局[1]
> 以及其面临的**地缘政治风险**与**应对措施**[4], 但未提供**海外业务**具体的
> **营收**数据."

函数匹配到 "海外" + "地缘政治" + "风险", 判"答对". 但答案开头 "根据已有资料无法回答"
已经说明这是拒答.

### 修法 (Day 10 开工时应用)

```python
# 在关键词匹配前, 先检测拒答语气 (优先级最高)
REFUSAL_PATTERNS = [
    "根据已有资料无法回答", "无法回答", "未提供", "未披露", "未提及",
    "资料不足", "参考资料中未", "参考资料未",
]

def check_answer(query_id, answer):
    # Step 1: 检测拒答态
    if any(p in answer for p in REFUSAL_PATTERNS):
        return False, "LLM 拒答"
    # Step 2: 原关键词逻辑
    ...
```

Day 10 开工前把这段加到脚本 28, 如果 Day 11 重跑大样本评估要用同样的答对判定,
这个修正必须入库.

## 4. 三个重大洞见

### 洞见 1: 金页标注策略系统性偏严 (TD-9-1b 第三次印证)

**U6 p.108 (审计报告复述页) 是答案的有效源, 但被我标成 false**.

这对 TD-9-1b "金页粒度是页级近似非 chunk 级精标" 是**第三次印证**:
- 第一次 (Day 9 Task 1 阶段 A): 用户提 "grep 太机械"
- 第二次 (Phase 1): U2 Top-20 命中 5/4 暴露同页多 chunk 问题
- 第三次 (Phase 2 U6): 非主披露页 (审计报告复述页) 也是有效答案源

**对 Day 11 RAGAS 评估的影响**:
- `context_recall / precision` 的 ground truth 定义要重新讨论
- "只标主披露页" vs "标所有含目标数字的页" 是两套不同的评估基线
- Day 11 评估前要和 TD-9-1b 一起系统决定

### 洞见 2: LLM 在"精确数字查询"上对数据噪声有韧性

U6 所有 10 个 mode 都能从 non-gold 页答出精确数字 **423,701,834 千元**.

**启示**:
- LLM 不需要完美金页, 召回 chunk 只要 **包含目标数字** 就够
- 这和 Day 7 Q4 "Reranker 能力边界" 是反面 — LLM 对数据噪声的容忍度比想象的高
- **"Top-K 金页命中率"作为评估指标有根本局限**: 低命中率 ≠ 答不对

**对 Day 10 的影响**: Day 10 Prompt 工程时要区分:
- **"检索失败类"查询** (LLM 真的找不到数字): Prompt 工程帮不上
- **"召回够用类"查询** (LLM 能从噪声里提取): 调整 Prompt 让 LLM 明确引用来源页

### 洞见 3: L2a 多口径错位扩展 — 从"招行特例"升级为"通用模式"

U3 的答案明确区分 "子公司净利润 427M" vs "集团合并净利润未提供":
```
"参考资料中提供了国电南自部分子公司2024年的净利润数据, 
例如南京国电南自自动化有限公司为427,419,723.66元,
南京国电南自维美德自动化有限公司为115,172,643.92元 [2].
但未提供国电南自母公司或整个集团合并口径的2024年度净利润总额."
```

**这是和 U4 招行同类的多口径错位问题**, 但**新变体**:
- U4: 同一报告里有 3 个口径 (本公司 0.93% / 集团 0.94% / 消费信贷 1.52%), LLM 不知道选哪个
- U3: 检索召回了**相关但口径不对**的页 (子公司财报 vs 集团财报), LLM 正确拒答

**TD-8-2 技术债范围扩大**:
- 原: 招行三口径错位特例
- 新: **通用的"子公司 vs 集团 / 不同时期 / 不同业务板块"多口径错位问题**

**Day 10 TD-8-2 Prompt 工程要处理的场景**:
1. 多口径并列时优先选集团合并口径 (U4 型)
2. 召回子公司口径但查询集团时, 不要答子公司数据 (U3 型) — 这个 **LLM 已经做对了**, 但 Prompt 可以帮 LLM 明说 "可以告诉用户: '已召回子公司数据, 但集团口径未找到'"
