## 0. 本日目标

Day 6 的 6 模式大对比实验明确暴露了"宽召回稀释"问题——Full 模式 6 个
探针召回 30+ 候选, RRF 融合后 Top-5 名额被"大致相关"页面占据, 精信号
在多路投票中被稀释。招行、紫金、茅台三个案例都出现了"更复杂方法反而
不如简单方法"的反直觉现象。

Day 7 的假设是: 用 **Cross-Encoder Reranker** 做"召回 + 精排"两阶段架构,
对 Hybrid/Full 的 Top-20 候选做细粒度重排, 把被稀释的精信号拎回 Top-5。
理论基础是 Cross-Encoder 能做 query-doc 交互式 attention, 判别力远强于
Bi-Encoder 的独立向量比较。

实际跑完 3 查询 × 3 模式 × 有无 Reranker = 18 组 A/B 对比后, 拿到三个
**反预期发现**:

1. **9 组 A/B 只有 2 组答案有可见提升**——Reranker 不是检索增强的银弹
2. **Reranker 的真实价值分 4 层**(排名救援 / 补证提升 / 无变化 / 数据锁死),
   "L1 排名救援成功 ≠ L2 答案质量提升"——Reranker 和 LLM 是两个独立瓶颈
3. **招行闭环实验意外挖出"Chunking 边界"问题**——p.45 chunk 里只写
   "报告期末"丢了"2025 年"锚点, 这是字符级 chunker 的结构性副作用,
   精排不能补救召回前的结构损伤

让我意识到: **工程实验最大价值不是验证预期, 是证伪预期**——如果每次实验
都符合想象, 那实验本身是多余的; Reranker 的真实价值光谱比课本上写得
窄得多, 但正是这个认知让我看清了它的边界。

## 3. 核心知识点

### 3.1 Bi-Encoder vs Cross-Encoder: 架构决定命运

你 Day 3 用的 BGE-small-zh 是 Bi-Encoder, 工作方式是:

```
query → BGE → [512 维 Q]
doc   → BGE → [512 维 D]       相似度 = cos(Q, D)
```

**两边独立编码**, 比较最终向量。这架构的决定性优点: **doc 向量可预计算**
——你 6881 条 chunk 的向量都提前入 Qdrant, 查询时毫秒级 ANN。

致命短板: **query 和 doc 从头到尾没对话**。各自被压缩进 512 维, 最后
只做一次 cosine。信息瓶颈 → 对"大致相关"判断够用, 对"哪个更相关"
细粒度判断开始露怯——**这就是 Day 6 宽召回稀释的机制根源**。

Cross-Encoder 把 query 和 doc **拼成一个输入**, 一起过 Transformer:

```
[CLS] 招商银行不良贷款率 [SEP] 2024年集团不良贷款率 0.93% [SEP]
     ↓ 每一层 self-attention 让 query token 和 doc token 互相看
     ↓
     输出 relevance_score (标量)
```

每层 attention 都在做 query↔doc 的交互判断。query 的"不良贷款率"能
直接 attend 到 doc 里的"0.93%", 做精细相关性判断。

**类比**:
- Bi-Encoder = 相亲前各自写简历, 最后比对
- Cross-Encoder = 两个人面对面坐下聊 10 分钟

代价: **doc 不能预计算, 每次查询对每个候选都要完整 forward 一遍**。
对 6881 条跑 Cross-Encoder, CPU 上 6~10 分钟——生产不可接受, 必须
配召回层用。

### 3.2 两阶段架构: 召回 → 精排的工程哲学

这是工业界 RAG 的标准架构:

```
用户 Query
  ↓
┌─ 阶段 1 召回 (Recall) ─────────────────┐
│ Hybrid (Dense + BM25 + RRF)           │  ← Bi-Encoder + 字面匹配
│ 从 6881 条挑 Top-20                    │    O(log N), 快, 粗
└───────────────────────────────────────┘
  ↓ 20 候选
┌─ 阶段 2 精排 (Rerank) ────────────────┐
│ Cross-Encoder 对 20 条逐条打分         │  ← query-doc 交互 attention
│ 重排后取 Top-5                         │    O(N), 慢, 准
└───────────────────────────────────────┘
  ↓ 5 精排结果
LLM 生成答案
```

**三条设计哲学**:

**哲学 1 - 分工(各用其所长)**:
- 召回层目标: high recall(别漏)
- 精排层目标: high precision@top-k(排准)
- 一个模型做不好两件事, 分工互补

**哲学 2 - 漏斗(越精方法处理越少数据)**:
- 6881 → 20 → 5, 每层降一个数量级, 允许后一层用更贵方法
- 这是所有多阶段系统的通用设计原则

**哲学 3 - 召回是精排的天花板**(面试必考):
- 召回没捞到金页, 精排再强也救不回
- **Day 7 紫金 p.59 案例完美印证**: A/B Top-5 都没 p.59, 不是精排失败,
  是 p.59 根本没进召回的 Top-20

### 3.3 BGE-Reranker-base 的关键工程细节

**(1) 输出是 sigmoid 概率, 不是 logits**

| 模型 | 输出 | 典型范围 |
|---|---|---|
| bge-reranker-base (你用的) | sigmoid 概率 | 0 ~ 1 |
| bge-reranker-large | sigmoid 概率 | 0 ~ 1 |
| bge-reranker-v2-m3 | 原始 logits | -10 ~ +10 |

排序逻辑完全通用(降序排即可), 但分布不同——打印日志、设置阈值、
写断言时要按实际分布理解。

**(2) max_length = 512 的截断陷阱**

底层 XLM-Roberta, 拼接后超过 512 token 被截断。对你的普通文本 chunk
一般不触发, 但 **表格 chunk** 可能被填满——这就是 Day 7 比亚迪 full
精排 15 秒的根因(TD-7-2)。

**(3) CrossEncoder 封装 vs 裸 transformers**

`sentence_transformers.CrossEncoder` 是 `transformers` 的高级封装,
自动处理 tokenizer、[CLS] query [SEP] doc [SEP] 拼接、batch 组装、
输出处理。裸用 transformers 要 30 行代码, CrossEncoder 一行解决。

**(4) CPU 性能基线**

单对打分约 100~150ms, 20 条 batch 理想情况 2~3 秒。但实测 4.5~15s
跨度, 因为 chunk 长度分布影响巨大(长 chunk 填满 512 比短 chunk 慢 5 倍)。

### 3.4 Reranker 的真实价值光谱 (4 层模型)

这是 Day 7 最重要的认知升级。**不要以为 Reranker 是"排得更准" 的银弹**
——它的真实价值可分 4 层, 不同查询停在不同层。

```
L1 排名救援  ← Reranker 识别被宽召回稀释的精信号, 拎回 Top-K
    └─ 判别依据: Cross-Encoder 的细粒度 query-doc 交互 attention
    └─ ✅ 案例: 比亚迪 full p.245 Rank 4 → Rank 1

L2 答案质量提升  ← 新进 Top-K 的页面携带更高信息密度, LLM 答更完整
    └─ 判别依据: 补证页面提供的新维度/口径/数据
    └─ ✅ 案例: 长江电力 "六座梯级电站" 口径细化

L3 换了补证答案不变  ← 新页面不坏也不特别好, LLM 不用
    └─ 原因: Top-1 已足够, 副证谁上 LLM 都不 care
    └─ ➖ 最常见: 9 组中 4 组属此

L4 数据缺失 / 召回未覆盖  ← Reranker 无能为力
    └─ L4a 数据根本不存在(比亚迪毛利率)
    └─ L4b 金页没进召回 Top-20(紫金 p.59)
    └─ L4c chunk 丢失上下文锚点(招行 p.45)
    └─ ❌ 这类问题需要上游修复
```

**核心洞察**:**L1 救援成功 ≠ L2 答案提升**

比亚迪 full 模式里 Reranker 完美完成了 L1 (p.245 从 Rank 4 → Rank 1),
**但答案仍然拒答**——因为 p.245 虽然"相关", 但里面没有原始披露数据。

**这证明 Reranker 和 LLM 是两个独立的瓶颈**:
- Reranker 管"把相关页排前", 完成它的任务
- LLM 管"从相关页里抽出答案", 依赖数据是否存在
- 前者赢不代表后者赢

### 3.5 Chunking 是 RAG 的最上游瓶颈 (招行案例启示)

Day 7 的闭环实验(scripts/23b)意外挖出一个 Day 5/6 都没发现的问题。

背景: Day 7 案例 1 原假设是 "Reranker 把 p.45 救到 Rank 1 才对"。
实验结果: **Reranker 反而把 p.45 挤出了 Top-5**。我的初判是 "p.45 是
2025 年数据, query 问 2024 年, Reranker 精读出年份错位降权正确"。

闭环验证: 把 query 改成 "招商银行 **2025 年** 不良贷款率", 两组都应
答对。

实际闭环结果意外:

```
A (无 rerank) Top-5: [50, 45, 5, 34, 35]   LLM 答: 拒答
  答案理由: "资料[2]仅说明'截至报告期末, 本公司不良贷款率为 0.93%',
           而报告期具体是哪一年并未在提供的片段中明确"

B (有 rerank) Top-5: [32, 35, 50, 33, 34]  LLM 答: 拒答
```

关键发现: **p.45 的 chunk 里根本没有"2025 年"字样**, 只写"报告期末"。
**这是 RecursiveCharacterTextSplitter 按字符长度切块的结构性副作用**
——"报告期 = 2025 年"的上下文信息被切到了其他 chunk, 留给 p.45 的
chunk 成了"悬浮数字"。

三层诊断递进:

| 层 | 诊断版本 |
|---|---|
| 表面 | Full 把 p.45 挤出 Top-5, 宽召回稀释, 上 Reranker 解 |
| 中层 | Reranker 把 p.45 降权是对的, p.45 是 2025 年数据口径错位 |
| 结构 | **p.45 chunk 里根本没年份锚点, chunker 把时间上下文切走了, 这是 chunking 结构性缺陷, 不是 Reranker 问题也不是数据边界** |

**工程原则**:
- **精排不能补救召回前的结构损伤**
- **检索优化不能补救 chunking 的信息损失**
- **RAG 性能 = chunker × embedder × retriever × reranker × LLM**, 最
  上游的 chunker 决定了下游的天花板

**缓解方案**(TD-7-1, Day 10 修):
- metadata 注入: chunker 阶段把 report_year 作为 metadata 存入每个
  chunk, Prompt 里拼入"参考资料来自 XX 公司 XX 年年报"
- 语义 chunking: 按章节/小节切而不是按字符切(Day 11+ 或项目外)


## 4. 关键实验发现

### 案例 1: 招行 2024 不良贷款率 — 从"Reranker 失败"到"Chunking 边界"的三层诊断

**场景**: 复现 Day 6 翻车案例, 期望 Reranker 修复宽召回稀释。

**结果表**:

| 步骤 | Top-5 | LLM 答案 | 诊断 |
|---|---|---|---|
| full OFF (Day 6 复现) | [45, 50, 47, 34, 32] | 拒答 | p.45 含 0.93% 但说"2025 年末" |
| full ON | [34, 31, 33, 35, 32] | 拒答 | p.45 被挤出 |
| **闭环 query 改 2025** | - | - | - |
| full OFF | [50, 45, 5, 34, 35] | 拒答 | p.45 含 0.93% 但"报告期末"无年份 |
| full ON | [32, 35, 50, 33, 34] | 拒答 | p.45 再次被挤 |

**三层诊断演进**:
1. **表面**: Reranker 把金页挤掉了 → 看起来是 Reranker 失败
2. **中层**: p.45 是 2025 年数据, query 问 2024, Reranker 降权正确
3. **结构(真相)**: **p.45 chunk 里根本没有"2025 年"字样**, 只写"报告期末"
   ——chunker 把年份上下文切到了其他 chunk

**工程价值**: 这个诊断让我看清了 RAG 的"结构性瓶颈"——精排只能对
"文本内已有信息"做细粒度判别, **恢复不了 chunker 切割时丢失的上下文**。
"L4c chunk 丢失上下文"比 L4a 数据缺失更隐蔽, 因为数据明明在库里, 只是
被切走了锚点。

**面试价值**: 展示"从症状到归因到结构"的三层诊断递进。**senior 和
junior 的分水岭是愿不愿意下到结构层**, junior 遇到问题先改代码/换
技术, senior 先分类归因。

### 案例 2: 比亚迪毛利率 — L1 救援成功 + L4 锁死的双重含义

**场景**: 查询"比亚迪 2025 年汽车业务毛利率", 3 模式 × 有无 Reranker。

**关键排名变化**(full 模式):

```
full OFF Top-5: [266, 209, 90, 245, 47]   → p.245 Rank 4
full ON  Top-5: [245, 118, 134, 90, 16]   → p.245 Rank 1 !!
```

**这是教科书级的 L1 排名救援**: Reranker 精读后把 p.245 从 Rank 4
拎到 Rank 1, 宽召回稀释被修复。

**但 3 模式 × A/B 答案全部拒答**:

> "参考资料中提到了比亚迪 2025 年汽车业务的收入约为人民币 648,646
> 百万元, 但未提供汽车业务单独的营业成本数据, 因此无法计算毛利率。"

**诊断**:
- p.245 相关, 但里面只有收入、没有对应成本
- 年报披露口径是"按产品/行业划分的收入"+ "集团合并成本", **没有
  "汽车业务单独成本"**
- 毛利率需要 (收入 - 成本) / 收入, 成本口径缺了, 算不出来

**结论**: Reranker 完美完成了 L1 救援, 但 L4 (数据不存在) 锁死了答案。

**双重含义**:
1. Reranker 工作正常, 宽召回稀释问题确实存在、也确实能被 Reranker 修
2. **"L1 赢了不等于 L2 赢"**, 这是 Reranker 的能力边界——它管排序,
   不管数据存不存在

**面试话术**: "检索优化不能弥补数据缺失, Reranker 再完美也变不出年报
没写过的数字。"——这句话简洁有力, 任何面试都能用。

### 案例 3: 长江电力发电量 — 最"干净"的 L2 救援

**场景**: 查询"长江电力 2024 年总发电量", 3 模式 × 有无 Reranker。

**关键排名变化**(hyde/full 模式同构):

```

hyde ON  Top-5: [8, 10, 11, 74, 25]     → p.11/74/25 新进

full OFF Top-5: [8, 10, 85, 60, 248]
full ON  Top-5: [8, 10, 11, 74, 25]     → p.11/74/25 新进(完全同 hyde)
```

**答案变化**:

```
OFF: "长江电力 2024 年总发电量为 2959.04 亿千瓦时"
ON : "长江电力 2024 年境内所属六座流域梯级电站的总发电量为
```

**诊断**:
- Top-1 (p.8) 不变, Reranker 没换主证

  流域梯级电站"这个业务口径细节
- LLM 引用 `[1][2][3][4][5]` 从 5 条都抽取信息, 答案因此加了口径细化

**这是 L2 救援的最纯粹形态**: 不是正确率提升(两边都答对了 2959),
是**答案负责任度提升**——"裸数字"升级为"带业务口径的数字"。

**面试价值**:
- **"Reranker 的主要价值不在翻 Top-1, 在补证质量"**——这是 Day 7
  跑 18 组 A/B 才看清的真相
- "答案负责任度"是 LLM 应用里常被忽视的维度——不是对错, 是"答得
  多严谨", Reranker 能提升这个

### 案例 4: 茅台营收 — L3 "Top-5 大洗牌但答案零变化"

**场景**: 查询"贵州茅台 2023 年营业收入", 3 模式 × 有无 Reranker。

**关键排名变化**(full 模式):

```
full ON  Top-5: [56, 20, 123, 14, 8]     → 除 Rank 1, 其他全换!!
```

## 5. 面试话术

### Q1: Cross-Encoder 和 Bi-Encoder 的核心区别? 为什么需要两阶段架构?

> "Bi-Encoder 和 Cross-Encoder 的根本区别在于 **计算图**。
>
> Bi-Encoder(我用的 BGE-small)是 query 和 doc **独立编码**后比较最终
> 向量——好处是 doc 向量可预计算, 我项目里 6881 条 chunk 都入 Qdrant
> 做 ANN, 查询毫秒级; 坏处是两边从头到尾没交互, 512 维向量作为最终
> 判断依据, 信息压缩严重, 对'大致相关'判断够, 对'哪个更相关'开始露怯。
>
> Cross-Encoder 是 query 和 doc **拼接成一个输入** 一起过 Transformer,
> 每层 self-attention 都让 query token 和 doc token 互相 attend。比如
> query 里的'不良贷款率'能直接 attend 到 doc 里的'0.93%', 做细粒度
> 判断。代价是 doc 不能预计算, 每次查询对每个候选都要完整 forward——
> 对 6881 条跑 Cross-Encoder 要几分钟, 生产不可接受。
>
> 所以必须是**两阶段架构**: Bi-Encoder 做召回层(快速从 6881 挑 20),
> Cross-Encoder 做精排层(对 20 条精细重排)。这有三条设计哲学——
> **分工**(各用其所长), **漏斗**(越精方法处理越少数据), 以及最
> 重要的一条——**召回是精排的天花板**: 召回没捞到的金页, 精排再强
> 也救不回。
>
> 这不是我抽象说的, 是 Day 7 紫金案例实锤过——Reranker 帮不上忙, 不
> 是精排无能, 是 p.59 根本没进召回 Top-20。"

**答题结构**: 架构差异 → 算力差异 → 漏斗设计 → 真实案例印证

### Q2: 你的 Reranker 实际收益怎么样? 延迟代价值得吗?

> - 0 组退化
>

> 用户绝对不能接受。
>
> 这让我认识到 Reranker 的**真实价值分 4 层**:
> - L1 排名救援(把宽召回稀释的金页拎回)——我验证了, 比亚迪 p.245

> - L2 答案质量提升(新进补证提升答案负责任度)——长江电力 hyde/full

> - L4 数据缺失或召回未覆盖——Reranker 无能为力
>

> Reranker 和 LLM 是两个独立瓶颈。
>
> 所以我的结论是: **Reranker 不应该默认开启, 应该条件开启**——只对
> '多维度查询'(LLM 会用多份互补证据)开, 纯事实查询不开。这又是
> 一个 Router 能解决的问题, 对应 Day 10 要做的'质量-延迟自适应策略'。"

**答题结构**: 诚实报告预期反转 → 4 层光谱模型 → 延迟数据 → 工程结论
(条件开启 + Day 10 路径)

**特别说明**: 这是 Day 7 最有分量的问题, 和 Day 6 Q5 的"宁德失败"类似,
都是"主动讲失败 + 展示诊断 + 升华到方法论"。

## 6. 已知问题 & 技术债

### 6.1 Day 7 新增技术债

#### TD-7-1 (P1): Chunking 边界丢失上下文锚点

**现状**:
招行 p.45 的 chunk 里只写"报告期末不良贷款率 0.93%", 没有"2025 年"
字样——RecursiveCharacterTextSplitter 按字符长度切块, 把"报告期 =
2025 年"的上下文切到了其他 chunk。

**影响**:
- 类似的"悬浮数字"chunk 让检索技术失效——Reranker 无法恢复切割时丢失
  的上下文
- 影响所有"XX 公司 XX 年 XX 指标"这类查询的召回质量
- 对精排无能为力, 是结构性问题

**影响范围**: 未知——Day 8 评估集上要量化"chunking 边界"触发比例

**缓解方案**:
- **Day 10 低成本方案**: chunker 阶段把 `report_year` 作为 metadata 注入
  每个 chunk, Prompt 里拼入"参考资料来自 XX 公司 XX 年年报"作为全局
  上下文——让 LLM 把缺失的时间锚点从 metadata 补回来
- **Day 11+ 根治方案**: 改用语义 chunking(按章节/小节切), 保证上下文
  完整性——但工程量大, 需要重建 Qdrant

**暂不立即修原因**: 需要重做 chunking + 重建 Qdrant, 不阻塞 Day 8-9 评估主线

---

#### TD-7-2 (P2): Reranker 延迟不稳定, chunk 长度是主因

**现状**:
精排耗时区间 4.5s ~ 15s, 3 倍跨度。最严重: 比亚迪 full + Reranker
端到端 28.13s。

**根因**:
BGE-Reranker-base 底层 XLM-Roberta max_length=512, 长表格 chunk 被
填满后单条 forward 比短 chunk 慢 5 倍。Top-20 里混入几个填满的长 chunk,
平均耗时就炸。

**影响**:
- 生产环境不可接受 28 秒延迟
- Reranker 延迟成为 RAG 端到端延迟的主要 bottleneck

**缓解方案**:
- 截断到 256 token 权衡速度精度(Day 9 评估后定)
- 对超过某阈值的长 chunk 跳过精排(直接用 Hybrid 排序)
- Router 决策是否开 Reranker(TD-7-4)

**计划修复时机**: Day 9 评估后根据数据决策

---

#### TD-7-3 (P2): Reranker 不做"精确口径"判别

**现状**:
Day 7 的最小 demo 中, A (整体 0.93%) vs B (房地产业 4.64%) 分数
0.998 vs 0.993——Cross-Encoder 判"都相关", 不判"是不是同一口径"。

**影响**:
- 口径错位的页面会同排 Top, 依赖 LLM 读内容区分
- 当 LLM 碰到多个口径混杂的候选, 可能选择拒答(而不是抽取其中一个正确的)

**缓解方案**:
- 这是 Cross-Encoder 的能力边界, 不能直接修
- 未来方向: 用 LLM-as-Judge 做二次过滤(比 Cross-Encoder 精细, 但贵很多)
- 或: 更精细的 Prompt 引导 LLM 识别口径一致性

**暂不修原因**: 能力边界问题, 需要单独架构(如 LLM-rerank)

---

#### TD-7-4 (P3): 默认开 Reranker 延迟不可接受, 需条件开启

**现状**:
Reranker 默认开启会让所有查询加 5~15s 延迟, 但 9 组中只有 2 组答案
有提升——收益稀薄。

**缓解方案**:
扩展 Query Router 决策是否开 Reranker:
- 多维度查询(如"紫金海外营收和风险"): 开(可能有 L2 补证增益)
- 纯事实查询(如"茅台 2023 营收"): 不开(Top-1 已对, 无增益)

**计划修复时机**: Day 10 "质量-延迟自适应策略"

### 6.4 重要认知升级: RAG 的瓶颈分层

Day 7 让我对 RAG 系统的瓶颈有了分层认知:

```
                  用户体验 = f(答案质量)
                          ↑
               ┌──────────┴──────────┐
               ↓                      ↓
            L2: LLM 作答            L4: 数据缺失
          (能不能抽出答案)         (数据存不存在)
               ↑
          L1: 召回 + 精排
         (金页有没有排进 Top-K)
               ↑
         L0: Chunking
       (金信息有没有被切碎)
```

从 L0 到 L4, **每一层的失败都会锁死下游**。Day 7 的四个案例分别命中
不同层:
- 招行案例命中 L0 (chunking 丢上下文)——再强的 L1/L2 也救不回
- 比亚迪案例命中 L4 (数据缺失)——L1 救援成功也白搭
- 长江电力案例命中 L1 被部分优化(L2 补证 → 答案质量提升)
- 茅台案例命中 L1 已经对了(Reranker 的 L1 增量为零)

**工程启示**: 优化要从 L0 开始, **上游损伤下游无法补**。这是 Day 8-10
路径设计的底层逻辑——Day 10 优先修 chunking 的缓解方案(TD-7-1), 不是
继续堆 Reranker 变体。
